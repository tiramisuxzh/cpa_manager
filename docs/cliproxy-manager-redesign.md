# CliProxyApi 额度池 / 文件池管理器重构方案

## 1. 重构前提

当前实现的问题不在于“颜色不好看”或“卡片不够多”，而在于产品边界已经偏离。

这个项目真正的核心目标是：

1. 代替命令行完成认证文件的查看、筛选、停用、启用、删除、上传。
2. 更准确地判断文件当前是可用、风险、401 失效、额度耗尽，还是其他异常。
3. 把高频批量动作做成可视化、可确认、可回溯的管理流程。

这个项目**不是**一个泛化的运维控制台，也不是一个“什么都能塞进去”的综合仪表盘。

因此，后续重构必须把产品重新收口为：

- `额度池管理`
- `文件池管理`
- `异常处理`
- `基础设置`

## 2. 设计原则

### 2.1 先列表，后卡片

- 主界面应以表格 / 列表为核心，而不是以大卡片和大段说明文案为核心。
- 用户进入页面后的第一目标是“定位文件并执行操作”，不是“先看一堆设计感组件”。

### 2.2 高危动作必须跟对象绑定

- 删除、停用、启用、批量处理都必须出现在和当前列表对象强相关的位置。
- 不能出现“文件勾选在 A 区域，删除按钮在 B 区域”的割裂交互。

### 2.3 额度视角和文件视角分开

- `额度池` 面向“账号当前还能不能跑、剩多少、何时重置、是否需要停用”。
- `文件池` 面向“文件本身是否启用、是否可删、是否要重新上传、是否要恢复”。
- 同一份数据允许两个视图展示，但不应该强迫用户在一个视图里同时理解两套语义。

### 2.4 只保留一个主滚动区

- 页面必须避免多层纵向滚动相互抢占。
- 每个主页面只允许一个主要滚动区，优先给表格主体。
- 侧边栏、页头、筛选栏都应尽量固定或轻量，不允许把列表挤出可视区。

### 2.5 设置类功能必须降级

- 连接地址、管理密钥、日志开关、代理、调试等，属于系统设置。
- 这些内容不能占用主工作台入口，更不能影响用户处理文件的第一屏效率。

## 3. 官方 API 对接边界

基于 `CliProxyApi` 官方管理 API，重构后分为两层能力。

### 3.1 P0 必须对接

这些接口直接服务于“额度池 / 文件池管理器”的主目标：

1. `GET /v0/management/auth-files`
   用于拉取认证文件列表，是整个管理器的数据入口。
2. `POST /v0/management/auth-files`
   用于上传认证文件。
3. `DELETE /v0/management/auth-files?name=...`
   用于删除文件。
4. `PATCH /v0/management/auth-files/status`
   用于停用 / 启用文件。
5. `POST /v0/management/api-call`
   用于按文件身份拉取真实额度信息。
6. `GET /v0/management/usage`
   用于补充 usage 成功 / 失败计数展示。
7. `GET /v0/management/usage-statistics-enabled`
   用于判断 usage 统计是否开启，决定界面是否展示相关列。

### 3.2 P1 可接入，但只能放设置页

这些接口有价值，但不应该进入首页主链路：

1. `GET/PATCH/DELETE /v0/management/proxy-url`
2. `GET/PATCH/DELETE /v0/management/request-log`
3. `GET/PATCH/DELETE /v0/management/logging-to-file`
4. `GET/PATCH/DELETE /v0/management/debug`
5. `GET/PATCH/DELETE /v0/management/request-retry`
6. `GET/PATCH/DELETE /v0/management/max-retry-interval`
7. `GET /v0/management/latest-version`

### 3.3 暂不进入第一期

这些能力不是当前产品核心，不应抢占研发重点：

1. `GET /v0/management/logs`
2. `POST /v0/management/vertex/import`
3. 其他迁移、导入导出、远程运维类接口

结论：
第一期先把“文件管理 + 额度判断 + 异常处理”做扎实，再考虑次级能力。

## 4. 产品信息架构

重构后的左侧菜单只保留 4 个一级入口：

1. `额度池`
2. `文件池`
3. `异常处理`
4. `设置`

不再保留“总览”作为默认首页。

原因：

- 用户打开工具后，第一诉求是处理文件，不是看总览。
- 总览天然是次级视图，适合做成 `额度池` 页顶部的一块轻量摘要，而不是单独霸占一级入口。

## 5. 页面结构定义

## 5.1 额度池

### 页面目标

- 快速看到当前可用账号的额度情况。
- 识别哪些账号额度耗尽但不该删除，只应停用观察。
- 为恢复操作提供数据依据。

### 页面布局

上到下固定为三层：

1. 轻量筛选条
2. 额度表格主体
3. 详情抽屉

### 筛选条字段

- 搜索：文件名 / 邮箱 / 套餐
- 状态筛选：全部 / 健康 / 风险 / 额度耗尽 / 已停用
- 剩余额度区间：`>50%` / `20%-50%` / `<20%` / `0%`
- 是否显示已停用

### 表格列

1. 勾选
2. 账号 / 文件名
3. 套餐
4. 会话剩余
5. 代码剩余
6. 重置时间
7. Usage 成功 / 失败
8. 当前判断
9. 最后刷新时间
10. 操作

### 批量动作

- 批量刷新额度
- 批量停用额度耗尽
- 批量移动到异常处理
- 批量启用已恢复文件

### 行级动作

- 刷新
- 查看详情
- 定位到文件池
- 停用 / 启用

## 5.2 文件池

### 页面目标

- 管理文件本身的生命周期。
- 解决“CLI 删除文件麻烦”的核心痛点。
- 让上传、启停、删除都集中在同一工作区完成。

### 页面布局

上到下固定为四层：

1. 上传区
2. 文件筛选条
3. 文件表格主体
4. 底部分页栏

### 上传区要求

- 只保留一个大区域。
- 整块区域支持拖拽和点击。
- 不再拆成“拖拽说明区 + 选择文件按钮”两套表达。

### 文件筛选条

- 搜索：文件名 / 邮箱 / 状态说明
- 文件状态：全部 / 启用中 / 已停用 / runtime_only / unavailable
- 文件分类：全部 / 正常 / 401 / 额度 / 其他异常
- 排序：最近刷新 / 文件名 / 异常优先

### 表格列

1. 勾选
2. 文件名
3. 账号
4. 提供方 / 套餐
5. 当前状态
6. 分类结果
7. 启用开关
8. 刷新时间
9. 操作

### 批量动作

- 删除选中
- 停用选中
- 启用选中
- 导出选中名称

### 行级动作

- 详情
- 启用 / 停用
- 删除
- 刷新额度
- 复制文件名

## 5.3 异常处理

### 页面目标

- 专门处理真正需要干预的对象。
- 把“401 应删除”和“额度应停用”明确分流。

### 页面布局

页面顶部用三段切换，不再散落在多个位置：

1. `401 异常`
2. `额度异常`
3. `其他异常`

### 每个分段下的动作规则

#### 401 异常

- 主动作：删除
- 次动作：详情、刷新、停用

#### 额度异常

- 主动作：停用
- 次动作：详情、刷新、启用观察
- 默认不提供“一键删除全部”作为首按钮

#### 其他异常

- 主动作：详情判断后再处理
- 允许删除，但需要明确确认

### 批量动作

- 删除 401
- 停用额度异常
- 删除其他异常
- 重新检测异常

结论：
异常处理页必须把“动作建议”做成页面规则，而不是把所有异常都统一交给一个删除按钮。

## 5.4 设置

### 页面目标

- 只承载低频配置。

### 包含内容

1. 管理地址
2. Management Key
3. 自动刷新间隔
4. 进入页面是否自动加载
5. 是否启用 usage 统计展示
6. 代理与重试策略

### 不应放在设置首页的内容

- 主列表
- 异常统计卡片
- 高频操作按钮

## 6. 分类规则重新定义

当前最大的业务问题不是“视觉不好”，而是“判断不够稳”。

后续分类必须明确分成三层：

### 6.1 一级状态

- `健康`
- `风险`
- `异常`
- `停用`

### 6.2 二级异常分类

- `401 认证异常`
- `额度异常`
- `其他异常`

### 6.3 动作建议

- `建议删除`
- `建议停用`
- `建议观察`
- `建议刷新后确认`

界面上必须同时展示“分类结果”和“动作建议”，不能只展示一个标签。

原因：

- 分类描述的是“它是什么问题”
- 动作建议描述的是“你现在该怎么处理”

这两者不是一回事。

## 7. 视觉与交互统一规则

## 7.1 字号层级

- 页面主标题：只在一级页面出现，且必须克制
- 区块标题：中号
- 表格内容：以 12px-14px 为主
- 禁止再出现“标题巨大、内容很少”的视觉浪费

## 7.2 色彩规则

- 健康：绿色系
- 风险：橙色系
- 401 / 硬异常：红色系
- 停用：灰紫或灰蓝

所有按钮、标签、状态点必须服从同一套语义映射，不能一个页面一套说法。

## 7.3 滚动规则

- 左侧菜单固定
- 页头固定
- 表格区域独立滚动
- 抽屉独立滚动
- 其他区域默认不再纵向滚动

## 7.4 反馈规则

- 行级按钮允许独立 loading
- 批量按钮允许独立 loading
- 一个按钮进入等待态时，不能把同一行其他按钮挤位
- 进度展示优先放在表格上方细条区域，而不是做成大面积卡片

## 8. 第一阶段落地范围

第一阶段只做下面这些，不再扩散：

1. 重做页面 IA
2. 重做 `额度池`
3. 重做 `文件池`
4. 重做 `异常处理`
5. 重做上传区
6. 收紧详情抽屉
7. 保留最少设置项
8. 修正分类逻辑和动作建议联动

第一阶段明确不做：

1. 远程日志主展示
2. 版本检测主展示
3. Vertex 导入
4. 复杂总览大屏
5. 花哨但无业务价值的装饰卡片

## 9. 建议的技术落地顺序

### 步骤 1

先砍掉当前三栏复杂布局，重建新的四菜单骨架。

### 步骤 2

优先重做 `文件池`，因为这是当前用户最痛的页面，也是删除、上传、启停的核心。

### 步骤 3

再做 `异常处理`，把 401、额度、其他异常真正拆开。

### 步骤 4

最后做 `额度池`，把额度视图做成以状态与恢复为核心，而不是重复文件池。

### 步骤 5

最后才补 `设置`，把连接和低频配置收纳进去。

## 10. 最终结论

这个管理器下一版不应该继续叫“控制台思路”的页面。

它应该是一套明确服务于以下目标的桌面型管理工具：

- 先找到文件
- 再判断状态
- 再批量处理
- 最后再看设置

如果后续继续重构，判断标准只有一个：

> 这个页面是否让用户比用 CLI 更快、更稳、更清楚地处理认证文件。

只要不能直接提升这个目标的内容，都应该降级、延后，或直接删除。
